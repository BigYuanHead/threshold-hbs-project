# 团队 Coding 规范

## 1. 文档目的
本规范用于统一我们 threshold hash-based signatures 项目的代码风格、协作方式和测试要求。

目标是：
- 保证代码正确
- 保证结构清晰
- 方便团队协作
- 方便测试和 benchmark 复现

---

## 2. 总体原则

1. **正确性优先**  
   先保证实现正确，再考虑性能优化。

2. **先做最小可运行版本**  
   先把基本流程跑通，再逐步加扩展功能。

3. **代码要可读**  
   组内每个人都应能看懂主要逻辑。

4. **重要参数不要写死**  
   所有关键参数应可配置。

5. **每个重要模块都必须可测试**  
   如果一个模块无法测试，就说明它还不够完整。

---

## 3. 命名规范

### 3.1 文件命名
使用小写字母 + 下划线。

示例：
- `lamport.py`
- `trusted_server.py`
- `bench_sign.py`

### 3.2 类命名
使用大驼峰命名法（PascalCase）。

示例：
- `LamportSignature`
- `TrustedServer`
- `UntrustedServer`

### 3.3 函数名和变量名
使用小写字母 + 下划线。

示例：
- `generate_keypair()`
- `sign_message()`
- `party_count`

### 3.4 常量名
使用全大写 + 下划线。

示例：
- `NUM_PARTIES`
- `THRESHOLD`
- `HASH_NAME`

---

## 4. 代码风格规范

1. **一个函数只做一件明确的事**  
   不要写一个函数把所有逻辑都塞进去。

2. **函数尽量简短**  
   如果一个函数太长，应拆成多个小函数。

3. **避免重复代码**  
   如果同样的逻辑出现多次，应提取成公共函数。

4. **变量命名要清晰**  
   避免随意使用 `a`、`b`、`tmp` 这类无意义名字，除非含义非常明确。

5. **不要保留无用代码**  
   push 前删除无用的 debug 代码、废弃代码和临时测试片段。
